# 01-02: DOM XSS em conversas.php

## Goal
Corrigir a vulnerabilidade critica de DOM XSS na linha 559 do `conversas.php` onde `text` do input do usuario e injetado diretamente em `innerHTML` sem sanitizacao, permitindo execucao de JavaScript arbitrario.

## MUST verify (blocking)
- [ ] Input do usuario no chat NAO usa `innerHTML` diretamente
- [ ] `textContent` ou sanitizacao aplicada no ponto de injecao

## Tasks

### T01: Corrigir DOM XSS no envio de mensagem manual

<read_first>
- `conversas.php` — ler linha ~559 onde `innerHTML` e usado com input do usuario
</read_first>

<action>
Localizar o codigo em `conversas.php` onde a mensagem do usuario e inserida no DOM.
O padrao atual e algo como:
```js
div.innerHTML = text + '<span class="msg-time">' + time + '</span>';
```

Alterar para:
```js
div.textContent = '';
const textNode = document.createTextNode(text);
const timeSpan = document.createElement('span');
timeSpan.className = 'msg-time';
timeSpan.textContent = time;
div.appendChild(textNode);
div.appendChild(timeSpan);
```

OU alternativamente usar `textContent` no container principal e tratar o time como elemento separado.
</action>

<acceptance_criteria>
- Nao existe `innerHTML` recebendo variavel `text` vinda de `input.value` em `conversas.php`
- `grep -n "innerHTML.*text\|innerHTML.*input\|innerHTML.*value" conversas.php` nao retorna matches com dados de usuario
</acceptance_criteria>

### T02: Verificar outros innerHTML com dados de usuario no mesmo arquivo

<read_first>
- `conversas.php` — revisar TODOS os usos de `innerHTML` no arquivo
</read_first>

<action>
Revisar todos os `innerHTML` no `conversas.php`:
- Linhas com `c.name`, `c.lastMsg`, `c.badge` (dados da API): substituir por `textContent`
- Linhas com `m.text` (corpo da mensagem da API): substituir por `textContent`
- Linhas com `ct.name`, `ct.phone`, `mem[k]` (CRM): substituir por `textContent`
- Linhas com `preview` (polling de preview): substituir por `textContent`

Para HTML que realmente precisa de formatacao (ex: `<b>`, `<i>` nas mensagens), usar DOMPurify ou uma funcao de escape que permite apenas tags seguras. Se nao houver necessidade de HTML rico, usar `textContent` puro.
</action>

<acceptance_criteria>
- `grep -c "innerHTML" conversas.php` retorna <= 2 (apenas usos seguros com strings hardcoded)
- Nenhum `innerHTML` recebe variavel vinda de API ou input do usuario
</acceptance_criteria>
